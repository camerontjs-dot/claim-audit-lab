## Phase Decision

Phase 13 was implemented as a small code-and-docs polish slice, not documentation-only.

Reason: the explicit anchor policy must be reflected in generated Markdown and locked by tests. The support-quality polish should also appear in the human-review report, because the current report exposes reliability/date/URL columns but does not yet make mixed or weak support differences prominent enough for reviewer scanning.

## Goals

1. Add an explicit Markdown anchor policy for report sections that need stable traceability.
2. Update Markdown rendering so claim, evidence-link, and rule-flag references are easy to locate from stable IDs.
3. Add report-level support-quality notes that make weak, unknown, stale, or mixed candidate evidence visible without changing final support labels.
4. Add tests that prove the anchor policy and support-quality notes are deterministic.
5. Refresh checked-in generated reports only after renderer tests define the intended output.

## Non-Goals

- No source discovery.
- No live LLM calls.
- No network calls.
- No support scores or assessment-confidence scores.
- No new claim labels unless a tested rule truly requires them.
- No research-use calibration, paired draft/final audit metrics, or production-data qualification.
- No Streamlit/FastAPI/UI work.

## Proposed Markdown Anchor Policy

Use explicit HTML anchors so links do not depend on a Markdown renderer's heading-slug rules.

Anchor IDs should be deterministic and derived from existing stable IDs:

- Report top: `report-{document_id}`
- Claim detail: `claim-{claim.id}`
- Rule flag: `rule-{flag.id}`
- Evidence link: `evidence-{source_id}-{excerpt_id}`

If an existing ID already includes a prefix, preserve the whole value rather than stripping it. For example, claim `claim-61e15c02b15c` should produce anchor `claim-claim-61e15c02b15c`. This is a little redundant, but it avoids silent ID rewriting and keeps the rule mechanical.

Anchor strings must:

- use lowercase ASCII
- replace characters outside `a-z`, `0-9`, and `-` with `-`
- collapse repeated dashes
- be tested against current fixture outputs

## Proposed Report Polish

Add a short support-quality surface to each claim detail when useful:

- weak or unknown reliability among direct supporting candidates
- stale direct support when `reference_date` makes freshness checks active
- mixed candidate quality where a high-ranked candidate and a lower-reliability or dated candidate both appear
- candidate evidence exists, but no direct support was found

This should remain explanatory. It must not turn candidate scores into final support labels or imply outside-world truth.

Suggested heading inside each claim detail:

```markdown
**Support quality notes**
```

If no special note exists, omit the subsection rather than adding filler.
